Introduction

Introduction

Hello and welcome to the Beginner's Guide of the UNIXverse! Today we will cover useful features available in shell - through the website content, talks and exercises (and hopefully, by the time we finished this, with solutions.

But first of all, what is shell? Many of you may think that the terminal you use (xterm, terminator, etc.) is a shell - you are partially right - it is an interactive shell. A shell is basically an interface that allows you to interact with a system's interface. The reason terminals you are used to are interactive is because you get to...well...interact with it (you give it input, they give output/errors, you give it more inputs [or rage quit].

One important thing to note about shells is...well, they are a "shell". All commands within a shell are self-contained - this will be an important concept in shell scripting.

Cryptic Symbols

When we are interacting with the terminal (or you are viewing commands online), you commonly see symbols like $, ~ and % (and most probably a bunch of text) before commands. This is merely to indicate that the command is entered by the user, and is not the output of a program. Below is a demonstration of what you may see.

# this is a comment
$ echo "you may see it like this"
you may see it like this
Oliver $ echo "or like this, with Oliver in front"
or like this, with Oliver in front
Oliver % echo "or this"
or this

A common newbie mistake when using shell is to include the preceding commands. Please don't do it, or you'll be a pleb, like the rest of the world who don't know shell (consider yourself upgraded!).

Shell Shortcuts

Here are a few (useful) shortcuts to shell.

Reuse previous command

Hitting the "up" key will automatically place a previously executed command in your next command to execute. Can do this as many times as you can go...

Going down, of course, does the opposite if you need it

Tab Completion

Essentially completes a command you have half-typed out so you don't have to waste precious milliseconds writing the rest of it. It is literally magical.

Mess around with the command above making directories and navigating through them so you get used to it all. Make a really long folder and then try out tab completion. An example might be:

mkdir antidisestablishmentarianism
cd anti<tab> <enter>

Try it, your life is now measurably better.

Standard Streams

This workshop sometimes relies on you knowing what the standard streams are - standard input (stdin), standard output (stdout) and standard error (stderr). They are further described below:

Standard Input

This is how data is read by the program - by user input unless it is redirected (see redirection). It is also known as stdin. You may know this as scanf in C or raw_input() in Python.

Standard Output

This is how data is output by the program - usually onto your terminal unless it is redirected. This is also known as stdout. Known as printf in C, or print in Python.

Standard Error

This is usually where all the error messages are written - which is also default to be written onto your terminal unless redirected. It is also known as stderr. You may know this as fprintf(STDERR, ""); in C or sys.stderr.write in Python.

Redirection

You may sometimes want to rewrite where the standard streams go - for example, read a file as standard output, or log errors from standard error into a file.

<

This places the standard input from a file into the program, for example, ./program < lol.txt

puts lol.txt into my program.

>

This places the standard output of a program into a file, for example, ./program > lol.txt

puts the output of my program into lol.txt.

You can use >> to append to the end, as > by default rewrites the entire program.

Note you can also combine redirectors: ./program < in.txt > out.txt.

Note you can use > /dev/null, which is the equivalent of putting all standard output into a sink of darkness and nothingness. This is useful for cutting out terminal spam without recording anything.

2>

This does the same redirection, but for standard error. For example, ./program 2> errors.txt logs errors from the program into error.txt.

You can also pipe standard error to standard input, for example, ./program 2>&1. This is useful for logging all errors and output to a file, such as ./program 2>&1 > lol.txt

|

This is a pipe, which takes the standard output of the left hand side and puts it into the standard output of the right hand side. This is very useful for the upcoming section on UNIX Filters, and is the bane of all commands. For example, ./program | ./program2 puts all output of program 1 into program 2.

Note all redirectors can be chained, for example, ./program < lol.txt | ./program2 | ./program3 > out.txt. A chain of pipes is known as a pipeline.
